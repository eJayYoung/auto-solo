# auto-solo 项目规划

## 需求背景

本项目面向飞书中的 Solo Coder 用户满意度标注流程，目标是把当前以人工为主的标注工作尽量半自动化，减少重复劳动，提高单日可处理题目数量，并降低字段填写和材料整理的出错率。

新的产品形态不再以零散脚本为主，而是以一个 `Next.js` 前后端一体应用来承接整个流程。用户使用飞书账号登录后，进入首页工作台，在一个统一系统里完成飞书同步、题目生成、仓库创建、任务编辑和飞书回填。

当前人工流程大致是：
1. AI 批量生成题目
2. 创建 GitHub 仓库并 clone 到本地，用 Trae 打开
3. 复制 prompt 到 Trae，选择测试模型后执行任务
4. 复制每轮会话的 Session ID、prompt、日志轨迹、产物截图
5. 填写飞书表格中的各类字段
6. 点击飞书表格里的 AI 质检并等待通过

从该文档可以归纳出几个关键约束：
- 题目不能过于简单，尤其不能是第一轮单文件代码理解类任务
- 每个多轮对话最多 5 轮
- 遇到死循环时要先重试一次，只提交重试后的结果
- 标注时要同时关注“任务是否完成”和“产物及过程是否满意”
- 不满意原因要具体，不能只写笼统结论

因此，这个项目的核心不是替代人工判断，而是把高重复、易出错的步骤自动化，形成一个适合批量执行的半自动 SOP。

## 项目目标

### 核心目标
- 构建一个基于 `Next.js` 的一体化工作台应用
- 使用飞书账号完成授权登录
- 首页集中承接飞书同步、题目生成、仓库创建和提交动作
- 提供任务表与题库表两套核心数据视图
- 支持任务表行内编辑与飞书回填
- 支持题库表 CRUD、批量生成、复制和状态管理
- 支持批量创建 GitHub 仓库、按需 clone 到本地并打开 Trae
- 保留人工判断与 Trae 内部执行环节

### 预期收益
- 提升题目处理效率
- 降低手工复制粘贴成本
- 减少表格字段漏填、错填
- 提高提交数据格式一致性
- 降低后续质检返工率
- 把原来分散在脚本和表格间的操作收敛到一个统一系统中

## 设计原则

1. **人工负责判断，系统负责承接标准化操作**
   - Trae 内的任务理解、继续轮次、满意度判断等保留人工
   - 同步、生成、建仓、字段整理、表格写入交给系统处理

2. **工作台优先**
   - 用户登录后直接进入首页工作台
   - 高频动作尽量一屏完成，减少多系统切换

3. **双表驱动**
   - 任务表承接飞书记录同步与反馈编辑
   - 题库表承接题目沉淀、生成、筛选和分发

4. **幂等与可重跑**
   - 创建仓库、同步飞书、回填字段等动作尽量支持重复执行
   - 某一步失败后不影响整条流程重新发起

5. **适配层隔离外部系统**
   - 飞书 Base、GitHub、Trae 等外部系统通过统一服务层接入
   - 避免业务逻辑直接耦合第三方接口细节

## 总体方案

项目采用“飞书登录 → 首页工作台 → 双表协作 → 工作台执行 → 飞书回填”的半自动流水线。

```text
飞书账号登录
  ↓
进入首页工作台
  ↓
一键同步飞书任务数据
  ↓
生成 / 管理题库
  ↓
一键创建 GitHub 仓库 / clone / 打开 Trae
  ↓
人工在 Trae 中执行任务
  ↓
在任务表中补充反馈和证据
  ↓
一键提交飞书表格
  ↓
人工点击 AI 质检
```

## 产品结构

### 1）首页工作台
首页是默认入口，用于承接高频操作和全局状态。

工作台应包含：
- 一键同步飞书表格
- 一键生成题目
- 一键创建仓库
- 一键提交飞书表格
- 今日待处理数量、同步状态、失败任务数等摘要信息

### 2）任务表
任务表来源于飞书多维表格同步，承接执行中的记录管理。

任务表需要支持：
- 同步飞书 Base 数据
- 按 UID、状态、轮次、任务类型筛选
- 行内编辑字段
- 会话分组展示
- 一键提交飞书表格

### 3）题库表
题库表来源于飞书多维表格中已经提交过的第一轮 `User Prompt`，承接题目沉淀与复用。

题库表需要支持：
- 同步历史第一轮题目
- 题目 CRUD
- 已提交题目不可删除
- 批量生成题目
- 一键复制题目
- 生成题目时绑定 `Trae Session ID` 为空的飞书记录
- 默认模型 `gpt5.4`

### 4）设置中心
设置中心承接用户级默认配置。

建议支持：
- 默认 GitHub 地址 / owner
- 仓库可见性
- 是否默认 clone 到本地，默认 `true`
- 是否默认打开 Trae，默认 `true`
- 默认本地目录，默认 `~/solo_projects`
- 默认题目生成模型，默认 `gpt5.4`

## 功能方案

### 一、飞书授权登录
系统采用飞书账号授权登录，作为整个应用的用户身份入口。

目标：
- 建立用户会话
- 绑定用户的飞书访问能力
- 为后续读取 Base、写回 Base 提供鉴权上下文

### 二、任务表同步与编辑
任务表是对飞书 Base 中任务记录的本地化工作视图。

需要支持：
- 首次同步飞书 Base 数据
- 增量刷新
- 行内编辑以下字段：
  - Trae Session ID
  - 轮次
  - User Prompt
  - 任务类型
  - 业务领域
  - 修改范围
  - 任务是否完成
  - 产物及过程是否满意
  - 不满意原因
  - github 地址
  - 分支/文件夹
  - 截图引用
  - 日志轨迹引用
- 一键提交飞书表格

### 三、题库表同步与生成
题库表既是历史题目沉淀区，也是新题目生产入口。

需要支持：
- 从历史已提交记录中抽取第一轮 `User Prompt`
- 对历史题目做去重沉淀
- 手动新增与编辑题目
- 删除未提交题目
- 批量生成新题目
- 对生成结果做重复和相似度校验
- 一键复制题目内容或完整 prompt

生成规则：
- 默认模型为 `gpt5.4`
- 支持批量生成
- 支持内置提示词追加模式
- 支持内置提示词覆盖模式
- 自动绑定飞书表格中 `Trae Session ID` 为空的记录

### 四、仓库创建流程
仓库创建由工作台发起，并服务于题目实际执行。

需要支持：
- 自定义 GitHub 地址 / owner
- 自定义仓库名称
- 是否 clone 到本地，默认 `true`
- 是否打开 Trae，默认 `true`
- 自定义本地保存路径
- 记录仓库地址、本地路径、分支等信息

可以参考已有脚本能力：
- `/Users/ejay/create-solo-repo.sh`
- `/Users/ejay/fill_lark_base_row.sh`

但在新系统中，它们更适合作为底层能力参考，而不是最终产品交互形态。

### 五、飞书回填流程
用户在 Trae 中完成测试后，回到系统中补充任务反馈，并提交回飞书 Base。

需要支持：
- 单条提交
- 批量提交
- 以 UID 精确定位目标记录
- 同步成功 / 失败状态展示
- 同步失败后保留草稿

## 实施阶段

### 第一阶段：应用底座
目标：先搭起 `Next.js` 应用骨架和登录能力。

范围：
- 初始化 Next.js 项目结构
- 飞书授权登录
- 用户会话管理
- 基础导航与首页工作台框架
- 通用配置与环境变量管理

### 第二阶段：任务表闭环
目标：让系统先具备飞书同步、查看和编辑任务的能力。

范围：
- 飞书 Base 拉取
- 任务表展示
- Session ID 分组
- 行内编辑
- 本地草稿保存
- 一键提交飞书表格

### 第三阶段：题库表闭环
目标：让系统承接题目沉淀、生成和绑定。

范围：
- 同步历史第一轮题目
- 题库 CRUD
- 批量生成题目
- 去重与相似度判断
- 自动绑定空 `Trae Session ID` 记录
- 一键复制

### 第四阶段：工作台执行闭环
目标：让系统从题目直接进入执行环境。

范围：
- 创建 GitHub 仓库
- clone 到本地
- 打开 Trae
- 保存仓库与题目关联关系
- 回填 GitHub 地址和分支信息

### 第五阶段：体验增强
目标：提升稳定性、可维护性和日常使用效率。

范围：
- 操作日志
- 错误追踪
- 审计记录
- 高级筛选与搜索
- 批量操作优化
- 同步状态看板

## 验证方案

### 1）登录验证
- 验证飞书账号可以正常登录
- 验证登录后可以访问工作台
- 验证用户会话失效后会正确跳转

### 2）任务表验证
- 验证飞书 Base 数据能正确同步
- 验证会话 ID 解析和分组逻辑
- 验证行内编辑可用
- 验证一键提交飞书表格成功

### 3）题库表验证
- 验证历史第一轮题目同步
- 验证题目 CRUD
- 验证已提交题目不可删除
- 验证批量生成题目功能
- 验证一键复制
- 验证绑定空 `Trae Session ID` 的逻辑

### 4）工作台验证
- 验证仓库创建
- 验证 clone 到本地
- 验证 Trae 能正确打开指定目录
- 验证 GitHub 地址 / owner 配置生效

### 5）回填验证
- 验证按 UID 定位飞书记录
- 验证本地编辑字段能同步回飞书
- 验证同步失败时草稿不会丢失
- 验证成功与失败状态能正确显示

## 风险与约束

1. **Trae 内部操作不宜完全自动化**
   - 模型选择、任务执行、是否继续轮次，仍然依赖人工判断

2. **飞书字段可能会调整**
   - 表格结构变化时，字段映射层要独立维护

3. **GitHub 与本地环境存在外部依赖**
   - `gh` 登录状态、本地目录权限、Trae 应用可用性都可能影响工作台动作

4. **相似度判断存在成本和阈值权衡**
   - 题目去重做得越强，模型调用成本越高

5. **题库与任务绑定必须可追踪**
   - 生成题目绑定了错误记录时，需要支持解除和重新分配

## 预期产出

本项目最终应形成：
- 一个稳定的 `Next.js` 全栈应用
- 一套飞书登录和飞书 Base 同步能力
- 一套任务表与题库表协同机制
- 一套可复用的仓库创建与执行准备流程
- 一套面向人工标注的半自动工作台
- 一套便于扩展的任务提交与归档机制

## 备注

本规划文档会随着流程落地持续迭代。当前优先目标是先把“飞书登录 + 工作台 + 任务表 + 题库表”这四段打通，再逐步补齐仓库创建、同步闭环和体验增强能力。
