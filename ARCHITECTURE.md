# auto-solo 架构设计文档

## 1. 文档背景

`auto-solo` 是一个面向 Solo Coder 用户满意度标注流程的 `Next.js` 全栈应用，目标是在不替代人工判断的前提下，把题目生成、仓库准备、飞书回填、反馈整理这些重复劳动标准化和系统化，从而提升单日有效产出。

当前人工流程主要包括：
1. 从题库生成题目
2. 创建 GitHub 仓库并 clone 到本地，打开 Trae
3. 将 prompt 复制到 Trae，选择测试模型后执行任务
4. 复制每轮会话的 Session ID、prompt、日志轨迹、产物截图
5. 填写飞书多维表格中的字段
6. 点击飞书表格中的 AI 质检并等待通过

相关依据如下：
- 标注规范文档：https://bcnrsnl3m9wk.feishu.cn/docx/XT4Hd11PeoUHGTxUSfjcY7XPnRb
- 飞书表格地址：https://bcnrsnl3m9wk.feishu.cn/base/HO1Kb3JQpa9e0WsHhQ0c2dpLnYb?table=tblfVRXx3qotQ4mO&view=vewKdKoVia
- 项目规划文档：`/Users/ejay/auto-solo/PLAN.md`

本架构文档用于把 `PLAN.md` 中的高层方案进一步细化为可实现的系统模块、数据模型、关键流程、集成边界与实施阶段。

## 2. 设计目标

### 2.1 核心目标
- 基于 `Next.js` 构建前后端一体的应用
- 使用飞书账号完成授权登录
- 首页即工作台，承接所有高频操作
- 提供任务表与题库表两套核心数据视图
- 支持一键同步飞书表格、一键生成题目、一键创建仓库、一键提交飞书表格
- 支持任务表行内编辑与本地草稿保存
- 支持题库表 CRUD、批量生成、复制与状态管理
- 降低人工复制、粘贴、整理和填表成本
- 降低因字段漏填、错填、重复题目带来的返工

### 2.2 系统定位
`auto-solo` 是一个**半自动化提效系统**，不是一个全自动标注机器人。它负责承接流程中的标准化环节，并为人工判断提供更高效的工作界面。

### 2.3 非目标
系统不负责以下事项：
- 不自动操作 Trae 内部的模型对话和执行过程
- 不自动替代人工判断“任务是否完成”“是否满意”等主观结论
- 不自动点击飞书 AI 质检按钮
- 不直接依赖飞书表格的固定 UI 结构，避免未来表结构调整时系统整体失效

## 3. 需求概述

系统需要支持以下四类关键能力：

### 3.1 任务表
- 从飞书多维表格同步历史提交记录
- 在本地任务表中按行展示并支持行内编辑
- 按会话 ID 分组展示相同会话
- 会话 ID 的解析规则：从 `Trae Session ID` 中提取第一个 `_` 与 `.` 之间的字符串
- 支持按 UID、状态、轮次、任务类型筛选
- 支持一键同步回飞书多维表格

### 3.2 题库表
- 同步飞书多维表格中已经提交过的第一轮 `User Prompt`
- 支持题目 CRUD
- 已提交过的题目不可删除
- 生成好的题目支持一键复制
- 支持批量生成题目
- 支持用户配置模型，默认 `gpt5.4`
- 生成题目时自动绑定飞书表格中 `Trae Session ID` 为空的记录
- 支持内置提示词 + 用户追加内容，或完全覆盖内置提示词
- 生成题目不得与历史题目相同或相似

### 3.3 工作台
- 首页即工作台
- 一键创建仓库、clone 到本地、打开 Trae
- 支持自定义 GitHub 地址 / owner
- 支持是否 clone 到本地，默认 `true`
- 支持是否打开 Trae，默认 `true`
- 支持自定义本地保存路径
- 默认路径为 `~/solo_projects`

### 3.4 飞书回填
- 用户开始用 Trae 测试模型后，在任务表中编辑反馈内容
- 支持一键提交飞书表格
- 同步时按 UID 精确定位

## 4. 总体架构

系统采用“`Next.js` 应用层 + 服务层 + 外部集成层”的结构。

```text
┌─────────────────────────────────────────────┐
│                 Next.js 应用                │
│  首页工作台 | 任务表 | 题库表 | 设置中心     │
└─────────────────────────────────────────────┘
                     │
                     ▼
┌─────────────────────────────────────────────┐
│                 服务层                      │
│  飞书同步服务 | 题目生成服务 | 去重服务     │
│  工作台服务   | 草稿服务     | 分组服务     │
└─────────────────────────────────────────────┘
                     │
                     ▼
┌─────────────────────────────────────────────┐
│                 集成层                      │
│  飞书 Base | GitHub | Trae | 本地文件系统   │
└─────────────────────────────────────────────┘
```

### 4.1 架构原则
1. 人工判断保留，系统只自动化标准流程
2. 本地草稿优先，飞书作为外部同步目标
3. 题目生成与历史数据去重前置
4. 所有外部动作可追踪、可重试、可恢复
5. 通过适配层隔离飞书 Base 和脚本细节

## 5. 产品模块设计

## 5.1 任务表模块

### 5.1.1 模块职责
- 从飞书 Base 读取历史记录
- 清洗为统一记录结构
- 解析会话 ID 并生成分组视图
- 提供按会话、按 UID、按状态的浏览能力
- 提供行内编辑与同步能力

### 5.1.2 展示形式
建议使用两层结构：
- **会话组视图**：按 `conversation_id` 分组
- **明细记录视图**：组内每一条记录按 UID / 轮次展示

### 5.1.3 会话 ID 解析规则
对于 `Trae Session ID`：
- 若为空，不参与会话分组
- 若存在，提取规则为：
  - 找到第一个 `_`
  - 找到其后的第一个 `.`
  - 取两者之间的字符串作为 `conversation_id`

例如：
```text
xxx_abc123.456
=> conversation_id = abc123
```

### 5.1.4 可编辑字段
在任务表中，以下字段应支持本地表单编辑：
- Trae Session ID
- 轮次
- User Prompt
- 任务类型
- 业务领域
- 修改范围
- 任务是否完成
- 产物及过程是否满意
- 不满意原因
- github 地址
- 分支/文件夹
- 截图引用
- 日志轨迹引用

### 5.1.5 不可编辑字段
以下字段应被视为只读或系统字段：
- UID
- 飞书 record_id
- 历史同步时间
- 历史提交状态
- AI 质检结果（如果系统只读取不控制）

---

## 5.2 题库表模块

### 5.2.1 模块职责
- 管理待执行题目
- 支持批量生成题目
- 管理题目状态
- 绑定待填写 UID
- 管理内置提示词与用户提示词模式

### 5.2.2 CRUD 规则
- **创建**：支持手动新增或 AI 批量生成
- **读取**：支持按状态、模型、业务领域、任务类型筛选
- **更新**：支持编辑题目内容、绑定 UID、修改备注
- **删除**：仅未提交题目可删除；已提交题目不可删除，只能归档或隐藏

### 5.2.3 批量生成能力
用户需要能配置：
- 生成模型，默认 `gpt5.4`
- 生成数量
- 业务范围 / 难度 / 类型限制
- 提示词模式：
  - 在内置提示词上追加内容
  - 完全覆盖内置提示词

### 5.2.4 内置提示词策略
系统内置提示词必须来自标注规范文档中的要求，至少包含：
- 题目不能过于简单
- 禁止第一轮单文件代码理解类题目
- 任务类型、业务领域、修改范围应合理分布
- 优先生成适合多文件、多模块、有明确工程目标的题目
- 不要和用户历史提交题目重复或近似

### 5.2.5 UID 自动绑定规则
系统在生成题目前，应先读取飞书 Base 中：
- `Trae Session ID` 为空的记录
- 将这些记录视为“待绑定 UID”池
- 新生成的题目按顺序或策略绑定 UID

### 5.2.6 一键复制
每个生成结果都需要支持：
- 复制题目内容
- 复制带上下文版本的 prompt
- 复制题目元信息摘要

## 5.4 同步中心模块

### 5.4.1 模块职责
- 管理本地草稿与飞书同步
- 校验字段完整性
- 按 UID 查找飞书记录并更新
- 显示同步任务状态

### 5.4.2 同步状态
每条记录建议有以下状态：
- `draft`：本地已编辑，未同步
- `syncing`：同步中
- `synced`：同步成功
- `failed`：同步失败
- `conflicted`：远端与本地存在冲突，需要人工确认

### 5.4.3 同步策略
- 以 UID 为业务定位键
- 以 `record_id` 为飞书更新键
- 允许单条同步与批量同步
- 同步失败后保留本地草稿
- 支持重试

### 5.4.4 可参考脚本能力
可参考 `/Users/ejay/fill_lark_base_row.sh` 中已有能力：
- 基于 UID 查找 Base 记录
- 通过 `lark-cli base +record-upsert` 更新字段
- 自动读取 git 远程地址和分支
- 规范化 GitHub URL

新系统应在此基础上扩展：
- 更完整的字段覆盖
- 批量同步
- 附件/截图/日志引用上传
- 同步日志和错误回显

## 7. 核心流程设计

## 7.1 飞书数据导入与会话分组

### 流程
1. 从飞书 Base 拉取记录
2. 映射为 `BaseRecord`
3. 对每条记录解析 `conversation_id`
4. 按 `conversation_id` 聚合
5. 生成任务表视图
6. 将 `Trae Session ID` 为空的 UID 放入“待绑定题目池”

### 关键点
- Session ID 为空的记录不参与会话聚合，但要参与 UID 待分配池
- 同一会话应支持展开查看各轮记录
- 若解析失败，应标记为“无有效会话 ID”并进入异常组

---

## 7.2 批量生成题目流程

### 流程
1. 用户设置模型，默认 `gpt5.4`
2. 用户选择生成数量
3. 用户选择提示词模式：追加或覆盖
4. 系统加载内置提示词模板
5. 系统读取历史提交题目作为去重基线
6. 系统生成候选题目
7. 候选题目进入相似度检测
8. 通过检测的题目绑定空 `Trae Session ID` 的 UID
9. 写入题库表，状态为 `ready`

### 关键点
- 若待绑定 UID 不足，应在生成前提示用户
- 若生成结果相似度过高，应自动重试或人工确认
- 题目与 UID 的绑定必须可追踪、可解除、可重新分配

---

## 7.3 工作台执行流程

### 流程
1. 用户从题库选择一个题目
2. 用户确认仓库名、本地路径、GitHub 配置
3. 系统调用 GitHub 创建仓库
4. 系统 clone 到本地目录
5. 系统打开 Trae
6. 系统保存 `WorkspaceProject`
7. 题库状态更新为 `running`

### 关键点
- 仓库名冲突时要允许重新命名
- 本地路径不存在时应允许自动创建
- 打开 Trae 失败时不能影响仓库创建结果记录

---

## 7.4 反馈编辑与同步流程

### 流程
1. 用户在 Trae 中完成部分或全部测试
2. 用户回到 auto-solo，在任务表中录入反馈
3. 系统写入 `FeedbackDraft`
4. 用户点击“一键提交飞书表格”
5. 系统根据 UID 查找飞书记录
6. 生成 `SyncJob`
7. 执行更新并返回逐条状态
8. 成功后草稿标记为 `synced`

### 关键点
- 本地编辑优先保存为草稿，防止误刷新或同步失败丢失
- 不满意原因等高价值文本字段要保留本地历史版本
- 支持单条同步和批量同步两种模式

## 8. 去重与相似度控制

为了避免生成与历史重复或相似的题目，建议采用分层去重机制。

### 8.1 第一层：精确匹配
- 与历史题目完全一致，直接拒绝
- 与已有题库中的待执行题目完全一致，也直接拒绝

### 8.2 第二层：归一化匹配
对文本做归一化后再比较，包括：
- 去除空格
- 去除标点
- 统一大小写
- 统一常见同义表达

若归一化后高度一致，则视为重复。

### 8.3 第三层：关键词重叠分析
比对以下维度：
- 技术栈
- 业务领域
- 任务类型
- 目标功能
- 修改范围

当多个关键维度同时高度重叠时，标记为高风险相似题目。

### 8.4 第四层：语义相似判断
使用模型或 embedding 做语义相似度计算：
- 超过阈值时拒绝或进入人工确认
- 阈值应允许在系统配置中调整

### 8.5 指纹持久化
系统应为每个已接受题目保存 fingerprint，用于未来生成时参与去重。

## 10. 外部集成设计

## 10.1 飞书 Base
系统通过飞书 Base 提供：
- 历史记录读取
- 空 `Trae Session ID` UID 获取
- 反馈字段同步更新
- 状态读取

设计上应通过一个 `lark-base adapter` 封装，避免后续飞书表结构变化直接影响业务层。

## 10.2 GitHub
系统通过 GitHub 提供：
- 仓库创建
- 仓库地址回填
- 仓库存在性校验

应通过 `github workspace service` 封装，而不是直接在 UI 层调用 shell。

## 10.3 本地文件系统
系统通过本地文件系统提供：
- 本地项目保存路径
- 草稿存储
- 缓存与日志存储
- 截图和日志文件引用管理

## 10.4 Trae
Trae 只作为外部启动目标：
- 系统负责打开项目目录
- 不接管 Trae 内部 prompt 输入与模型交互

## 10.5 AI 模型服务
AI 模型用于：
- 题目批量生成
- 题目相似度判定

默认模型配置为 `gpt5.4`，但应允许用户扩展或切换。

## 14. 风险与约束

### 14.1 飞书表结构变化
飞书 Base 字段可能调整，必须通过字段映射层隔离。

### 14.2 Session ID 格式不稳定
如果 Trae Session ID 格式存在变化，会影响会话分组逻辑，需要保留异常兜底。

### 14.3 相似度判断成本
语义去重能力越强，调用模型的成本越高，需要在准确率与成本之间平衡。

### 14.4 外部依赖不稳定
GitHub、飞书、Trae 启动、本地路径权限等都可能成为失败点，系统必须提供明确错误提示与重试能力。

### 14.5 人工流程仍然存在
系统只能提效，不能完全消除人工判断与人工采集，因此界面与流程设计必须尽量减少切换成本。

---

## 15. 结论

`auto-solo` 的最佳落地方向是：围绕“任务表、题库表、工作台、同步中心”四个核心模块，建立一个以本地草稿和结构化任务为中心、以飞书 Base 为同步目标、以 GitHub 和 Trae 为执行支撑的 `Next.js` 半自动化系统。

它的价值不在于完全替代人工，而在于把高频、机械、易错的动作统一进系统，从而让用户把时间集中在题目判断、模型执行和结果评估上。
